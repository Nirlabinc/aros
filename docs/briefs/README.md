## Why this package exists

On **2026-07-23** AROS chat broke on the founder's own production account: six of
twelve nightly eval questions returned `HTTP 401 {"error":"Invalid or expired
passport","code":"INVALID_TOKEN"}`, escalating to a total login failure by 08:47Z.
**Nothing alerted.** The nightly chat-eval watchdog — the only monitoring that
exists — errored, its triage crashed on the missing `results.jsonl`, its runner
threw away every exit code it collected, and Windows Task Scheduler recorded
`LastTaskResult: 0` (SUCCESS). A full-day production outage told nobody. When the
founder went looking for evidence afterwards, there was none: AROS keeps **no
server-side record of any conversation** — every turn is answered and thrown away.
The root cause turned out to be a missing `else` branch in the `/v1/*` proxy
(`src/server.ts:980-1004`) that forwards a Supabase token it knows the router will
reject. This package fixes the outage's cause (B), makes the system able to see
itself (A, E, F), makes its answers honest (C, D, I), and lands the two retail
profile apps that the same observability layer unblocks (G, H).

**Read first, in this order (≈10 minutes):**

1. `EVIDENCE-401-root-cause.md` — the mechanism behind the outage. Track B is
   designed against it; C and D inherit the honesty problem it creates.
2. `COORDINATION-ai-activity-spine.md` — **a concurrent session owns an
   overlapping mission.** Track A is the AROS half of their AI activity spine and
   must BIND to their actor-stamp contract, not invent a second one. It also lists
   files that are hands-off, and establishes that AROS has never billed a dollar of
   AI usage → **treat `cost_events` as EMPTY** in every track.
3. The **Blocking corrections** table below. Several briefs contradict each other.
